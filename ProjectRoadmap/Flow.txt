2️⃣ Security Layer (Authorization Architecture)

Rất nhiều hệ thống chết vì không rõ tầng này.

A. Role Model (đã định nghĩa rõ)

student

teacher

admin

B. Bảo vệ ở 3 cấp
Level 1 – Constraint cấp DB

Ví dụ:

CHECK (grade_level BETWEEN 6 AND 12)
CHECK (difficulty BETWEEN 1 AND 5)
NOT NULL
FOREIGN KEY

👉 Không phụ thuộc frontend.

Level 2 – RLS (Row Level Security)

Ví dụ:

student chỉ thấy exam của mình

teacher thấy tất cả exam

admin full access

👉 Bảo vệ data layer.

Level 3 – RPC Gatekeeping

Ví dụ:

assert_admin_or_teacher()

deactivate_question()

create_exam_from_template()

👉 Không cho frontend đụng raw table trực tiếp.

II. BACKEND LÀ GÌ TRONG PROJECT NÀY?

Với Supabase, backend của bạn gồm:

1️⃣ Database schema
2️⃣ RLS policies
3️⃣ RPC functions (business logic layer)
4️⃣ Trigger / snapshot logic
5️⃣ Constraint / Index

Nói cách khác:

Backend = Business logic nằm trong PostgreSQL

Frontend chỉ gọi RPC.

IV. DATA FLOW (QUAN TRỌNG)

Hiện tại hệ thống bạn có 3 loại dữ liệu:

1️⃣ Source of Truth (Question Bank)

Editable.

2️⃣ Snapshot (Exam Instance)

Immutable sau khi submit.

3️⃣ User Interaction

user_answers

Flow đúng phải là:

Question → Template → Exam Instance → Snapshot → Submit → Lock

Nếu snapshot không rõ → hệ thống vỡ.
